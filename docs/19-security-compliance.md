# Безопасность и соответствие 152-ФЗ

> **Версия:** v0.4.12 (2026-07-20)
> **Для кого:** админ, безопасник, юрист, руководство
> **Нормативка:** 152-ФЗ «О персональных данных», ФСБ/ФСТЭК требования к ИСПДн

---

## Что хранится в системе

### Категории данных

| Категория | Примеры | Класс ПДн по 152-ФЗ |
|---|---|---|
| **Технологические** | Обозначение детали, наименование, размеры, масса, материал, маршрут обработки | **Не ПДн** (производственные данные) |
| **Технические (о системе)** | Роль пользователя (cookie), IP, user-agent | **Не ПДн** (не идентифицирует личность) |
| **Справочные** | Наименования материалов, оборудования, ГОСТов | **Не ПДн** |
| **Идентификационные** | ФИО пользователей (если будут в `pilot_users`) | **ПДн** (категория — иные) |
| **Аудит действий** | Кто (роль) что сделал, когда | **ПДн** (если связано с ФИО) |

**Важно:** на текущей версии (v0.4.12) ФИО **не хранятся** — только роль (technologist, admin и т.д.). Это **не идентифицирует** конкретного человека.

Если в будущем добавим авторизацию по логину/паролю с ФИО — потребуется:
- Уведомление Роскомнадзора (если > 1000 субъектов)
- Согласие на обработку
- Модель угроз
- Сертификация (если требуется)

---

## Что мы НЕ храним (сознательно)

- ❌ **ФИО пользователей** (на пилоте — только роли)
- ❌ **Паспортные данные**
- ❌ **СНИЛС, ИНН**
- ❌ **Адреса, телефоны**
- ❌ **E-mail сотрудников** (в настройках — только e-mail админа для алертов)
- ❌ **Содержание чертежей** (только путь к файлу)
- ❌ **Фотографии сотрудников**

**Принцип минимизации:** храним только то, что нужно для работы.

---

## Класс ИСПДн

По 152-ФЗ (ст. 19) система обрабатывает:

- **Иные ПДн** (роль пользователя в cookie) — **УЗ-3** (упрощённый уровень защиты)
- **Без специальных категорий** (здоровье, религия, политика и т.д.)
- **Без биометрии**
- **Без трансграничной передачи**

**Модель угроз:** угрозы 3-го типа (УЗ-3) — несанкционированный доступ, утечка, утрата.

**Сертификация:** не требуется (УЗ-3 < 1000 субъектов).

**Уведомление Роскомнадзора:** не требуется (нет ПДн, идентифицирующих субъекта).

## Технические меры защиты

### 1. Аутентификация (на пилоте)

**Сейчас:** без пароля, только выбор роли через UI (cookie `bit_role`).

**Риск:** если злоумышленник получит доступ к URL — он может сменить роль.

**Митигация:**
- On-premise deployment (внутри ЛВС Техинкома)
- Доступ к серверу только изнутри сети завода
- Reverse proxy (Nginx) с IP whitelist (опционально)
- В будущем — LDAP/AD интеграция

**Планируется (v0.5+):**
- Логин/пароль (bcrypt + sha256+salt)
- LDAP/AD интеграция
- 2FA (опционально)

---

### 2. Авторизация (RBAC)

**Реализовано:**
- 7 ролей с разными правами
- API endpoints проверяют роль (с v0.4.10+)
- UI скрывает кнопки для неподходящих ролей

**Критичные endpoints** (требуют `technologist`, `main_technologist`, или `admin`):
- `POST /api/analyze` — генерация
- `POST /api/draft-fast` — генерация
- `POST /api/refine` — генерация
- `POST /api/generate` — генерация
- `POST /api/details` — создание детали (тоже)

**Admin-only:**
- `/admin/*` (8 страниц)
- `POST /api/admin/backup`

**Матрица прав** — см. [`14-roles-user-guide.md`](14-roles-user-guide.md) § Сводная таблица прав.

---

### 3. CSRF (Cross-Site Request Forgery)

**Включен по умолчанию** (с v0.4.10+).

Все POST/PUT/DELETE запросы требуют валидный CSRF-токен.

**Отключение:** `PILOT_CSRF_DISABLED=true` в `.env` (только для тестов!).

---

### 4. CSP (Content Security Policy)

Middleware добавляет заголовки:
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
```

Защита от XSS и clickjacking.

---

### 5. Rate limiting (in-memory)

Ограничения:
- `/api/generate` — 10/мин
- `/api/import/tk` — 5/5мин
- `/api/admin/backup` — 1/час

Защита от брутфорса и спама.

**Отключение:** `PILOT_RATELIMIT_DISABLED=true` (только для тестов!).

---

### 6. Шифрование секретов (Fernet)

`app_settings` хранятся в БД **зашифрованными** через Fernet (AES-128-CBC + HMAC-SHA256).

Ключ шифрования — в файле `.master_key` (вне БД, в `.gitignore`).

**Восстановление:** если `.master_key` потерян — все настройки придётся ввести заново.

**Резервная копия:** `.master_key` **обязательно** бэкапить отдельно (например, в password manager).

---

### 7. Шифрование паролей (bcrypt + sha256+salt)

Если в будущем добавим авторизацию:
- `bcrypt(sha256(salt + password))` — двойное хеширование
- Salt — 32 hex символа (cryptographically random)
- В БД хранится только `password_hash` + `salt`

---

### 8. Шифрование бэкапов (gpg)

`backup.sh` поддерживает gpg-шифрование:
```bash
export BACKUP_GPG_RECIPIENT="admin@tehincom.ru"
bash backup.sh
```

Бэкап шифруется публичным ключом админа. Расшифровать можно только приватным.

**Если не настроено** — бэкап **не зашифрован** (warning в `backup.sh`).

---

### 9. Audit log

Все действия записываются:
- `audit_logins` — кто входил (роль, IP, user-agent, успех)
- `history` — действия с деталями (кто, что, когда)
- `llm_calls` — все вызовы AI (деталь, стоимость, ошибка)

**Срок хранения:**
- `audit_logins` — 180 дней
- `llm_calls` — 90 дней
- `history` — 365 дней

Очистка через `cleanup_old_records()` (ежедневно cron).

## Организационные меры

### Доступ к серверу

- **Физический доступ** — серверная Техинкома, охраняемая
- **Сетевой доступ** — только из ЛВС Техинкома (или VPN)
- **SSH-доступ** — по ключу (рекомендуется) или паролю (для on-premise допустимо)

### Доступ к приложению

- **Админ** — знает пароль от `.env` и `.master_key`
- **Разработчик** — знает пароль от git-репозитория и SSH к серверу
- **Пользователи** — работают через UI, не имеют доступа к серверу

### Журнал изменений

Все изменения кода — в git (`bit-technolog-prototype`). История коммитов, авторы, даты.

### Incident response (при утечке)

**Если обнаружена утечка ПДн** (например, расшарили URL с конфиденциальными данными):

1. **Немедленно** (в течение 1 часа):
   - Закрыть доступ (сменить cookie, пароли)
   - Сделать бэкап текущего состояния (для расследования)
2. **В течение 24 часов:**
   - Уведомить руководство Техинкома
   - Уведомить куратора (Сергея Жукова)
   - Оценить масштаб (сколько субъектов, какие данные)
3. **В течение 72 часов:**
   - Если утечка серьёзная — уведомить Роскомнадзор
   - Уведомить субъектов ПДн (если их можно идентифицировать)
4. **Пост-фактум:**
   - Внутреннее расследование
   - Ужесточение мер (2FA, аудит, шифрование)
   - Публичный отчёт (если требуется)

**Контакты для incident response:**
- Админ Техинкома: см. контакты
- Куратор (Сергей Жуков): Telegram
- Роскомнадзор: https://rkn.gov.ru/, +7 (495) 987-68-00

## Retention policy (сроки хранения)

| Данные | Срок | Обоснование |
|---|---|---|
| `details`, `drafts`, `draft_versions` | **∞ (пока есть место)** | Производственная необходимость |
| `edits`, `rules` | **∞** | Производственная необходимость |
| `history` | **365 дней** | 152-ФЗ ст. 5 п. 5 (не дольше, чем нужно для цели) |
| `audit_logins` | **180 дней** | Безопасность (расследование инцидентов) |
| `llm_calls` | **90 дней** | Экономика и метрики (не ПДн) |
| `deleted_operations` | **365 дней** | Возможность восстановления |
| `app_settings` | **∞** | Пока система работает |
| `pilot_users` | **Пока активен + 90 дней** | Удаление неактивных |

**Очистка:** `cleanup_old_records()` в `admin.py`, вызывается ежедневно cron'ом.

**Ручная очистка:**
```python
cd /opt/beget/bit-technolog
./venv/bin/python -c "
from admin import cleanup_old_records
print(cleanup_old_records())
# {'audit_logins_deleted': 5, 'llm_calls_deleted': 23, 'history_deleted': 12}
"
```

## Права субъектов ПДн (по 152-ФЗ)

Хотя на текущей версии мы **не храним ПДн**, идентифицирующие субъекта, мы готовы к запросам:

- **Право на доступ** (ст. 14) — субъект может запросить свои данные
- **Право на уточнение** (ст. 21) — субъект может потребовать исправить
- **Право на удаление** (ст. 21) — субъект может потребовать удалить
- **Право на отзыв согласия** (ст. 9) — если обработка по согласию

**Контакт для субъектов:** admin@tehincom.ru (настроен в `/admin/settings` → `smtp_from`).

## Если расширите систему (добавите ФИО)

При добавлении ФИО, e-mail, телефонов потребуется:

1. **Согласие на обработку** (письменное, по форме)
2. **Уведомление Роскомнадзора** (если > 1000 субъектов)
3. **Положение о ПДн** (локальный акт организации)
4. **Модель угроз** (для ИСПДн класса УЗ-1/УЗ-2)
5. **Сертификация СЗИ** (для УЗ-1, государственные ИС)
6. **DLP-система** (для контроля утечек)
7. **Соглашение с оператором** (если данные передаются)

**Рекомендация:** для пилота (27 июля) — **НЕ добавлять ФИО**. Использовать только роли.

## Чеклист для админа (перед пилотом)

- [ ] Сменить дефолтные пароли в `.env`
- [ ] Удалить `.master_key` из git (он в `.gitignore` ✓)
- [ ] Настроить `BACKUP_GPG_RECIPIENT` для шифрования бэкапов
- [ ] Настроить Telegram алерты (для `/health`, бэкапов, ошибок)
- [ ] Проверить, что cron jobs работают (backup, verify, weekly, health, cleanup)
- [ ] Проверить, что `/admin/errors` пустой (нет критичных ошибок)
- [ ] Проверить, что `PILOT_AUTH_DISABLED=true` (для пилота без пароля)
- [ ] Проверить, что `PILOT_CSRF_DISABLED=false` (CSRF включён)
- [ ] Проверить, что `PILOT_RATELIMIT_DISABLED=false` (rate limit включён)
- [ ] Проверить, что firewall открыт только нужные порты (8081, SSH)
- [ ] Проверить, что БД бэкапится (минимум 1 бэкап есть)
- [ ] Проверить, что логи не растут бесконтрольно (logrotate работает)
- [ ] Проверить, что `PILOT_RATELIMIT_DISABLED=false` (не оставлять выключенным в production)
- [ ] Создать Telegram-чат для алертов (если не создан)
- [ ] Документировать процедуру восстановления (для incident response)
- [ ] Обучить пользователей работе с системой (5 мин вводный курс)
