## 1. Прямые конкуренты (стиль, отраслевой контекст)

### 1.1. ИИ-Технолог (tab-is.ru)

**Что делает:**
- Распознаёт чертежи любой сложности (геометрия, размеры, допуски)
- За 1-5 минут анализирует чертёж, рассчитывает трудоёмкость, формирует ТК + себестоимость
- Интегрируется с 1С, работает локально (on-premise)
- Заявленная точность +35% к ручному нормированию

**Стиль (по описанию и типичной 1С-экосистеме):**
- Enterprise-look, «1С-стиль»: тяжёлые формы, длинные выпадающие списки
- Минимум визуальной иерархии, всё одинакового размера
- Таблицы на весь экран с серым фоном
- Мало пустого пространства, плотная информация
- **Антипаттерн для нас** — наша задача быть красивее

**Что НЕ брать:**
- ❌ Тяжёлые формы с 30+ полей
- ❌ Серые таблицы без статусов
- ❌ 1С-цвета (зелёный/жёлтый без семантики)
- ❌ Спрятанные настройки в 5 уровней меню

**Что взять (если получится найти их UI):**
- ✅ Плотная информация (без whitespace-моды)
- ✅ Таблицы с возможностью сортировки по любой колонке
- ✅ Интеграция с 1С (у нас уже есть)
- ✅ On-premise (у нас уже есть)

**Скриншоты не нашёл публично** — у tab-is.ru демо по четвергам, надо сходить и заскринить.

## 2. Смежные продукты (юзабилити AI-инструментов)

### 2.1. Cursor (cursor.com) — AI-редактор кода

**Что делает хорошо (что взять):**

#### 2.1.1. Inline-edit через Cmd+K
- **Паттерн:** выделил текст → Cmd+K → написал что изменить → видишь diff → apply
- **Почему хорошо:** AI встроен В контекст работы, а не отдельный блок
- **Наше применение:** при редактировании операции — inline-кнопка «🤖 AI-помощь» прямо в поле, не отдельный AI-экран

#### 2.1.2. Composer через Cmd+I
- **Паттерн:** большая задача (multi-file) — открывается Composer panel сбоку
- **Почему хорошо:** пользователь не уходит со страницы, видит и свой код, и AI-работу
- **Наше применение:** AI-блок справа на экране детали (когда деталь уже сгенерирована — для уточнений)

#### 2.1.3. Apply buttons
- **Паттерн:** AI предложил изменения → одна большая зелёная кнопка «Apply» / красная «Reject»
- **Почему хорошо:** пользователь явно контролирует что применить
- **Наше применение:** после генерации ТК — «✅ Применить (5 операций)» vs «↩️ Отклонить» (у нас уже есть похожее через approve)

#### 2.1.4. Diff before apply
- **Паттерн:** AI показывает изменения inline с подсветкой (зелёный = добавлено, красный = удалено)
- **Почему хорошо:** пользователь видит ЧТО изменится до применения
- **Наше применение:** показать diff между старой ТК и AI-предложением (планируется в F14)

#### 2.1.5. Project rules (AGENTS.md)
- **Паттерн:** файл с правилами в корне — AI читает и следует
- **Почему хорошо:** «правила технолога» (наш `tech_rules`) — та же идея, мы это уже сделали

**Что НЕ брать:**
- ❌ Tab-ы между чатами (у нас нет multi-chat)
- ❌ Тёмная тема as default (наши пользователи — завод, 50+ лет, нужен светлый)

### 2.2. Linear (linear.app) — issue tracker

**Что делает хорошо:**

#### 2.2.1. Command palette (Cmd+K) — universal entry point
- **Паттерн:** Cmd+K → fuzzy search по всему (issues, projects, commands)
- **Почему хорошо:** одна точка входа для всего, не нужно искать в меню
- **Наше применение:** **ПРИОРИТЕТ #1 для M24** — добавить Cmd+K (Ctrl+K) command palette с поиском по деталям, командам, переходу к частым страницам

#### 2.2.2. Mnemonic shortcuts
- **Паттерн:** S = Status, P = Priority, A = Assignee
- **Почему хорошо:** легко запомнить, не нужно тренироваться
- **Наше применение:** S = переключить статус, U = утвердить, R = вернуть в работу

#### 2.2.3. Speed is a feature
- **Паттерн:** sub-100ms interactions (они публикуют метрики)
- **Почему хорошо:** perceived performance = emotional response
- **Наше применение:** все клики → мгновенный feedback (toast, optimistic update)

#### 2.2.4. Information density > whitespace
- **Паттерн:** show more data with less chrome
- **Почему хорошо:** опытные пользователи видят больше без скролла
- **Наше применение:** таблица деталей — все 7 колонок видны (✅ уже сделано в M22)

#### 2.2.5. Optimistic UI
- **Паттерн:** обновить локально → синхронизировать в фоне → ошибка только если реально fail
- **Почему хорошо:** UI не «лагает» из-за сети
- **Наше применение:** approve/reopen сразу обновляют UI, не ждут 200 OK

**Что НЕ брать:**
- ❌ Dark mode as default
- ❌ Минималистичный design (нашим пользователям нужны подсказки)

### 2.3. v0.dev / Bolt.new — генерация UI по промпту

**Что делает хорошо:**

#### 2.3.1. Progress-bar во время генерации
- **Паттерн:** показывает % выполнения + текущий шаг («Анализирую…», «Генерирую…», «Проверяю…»)
- **Почему хорошо:** пользователь видит что процесс идёт, не нервничает
- **Наше применение:** **ПРИОРИТЕТ #2 для M24** — добавить progress-bar для /api/draft-fast и /api/generate (у нас есть HTML, но он не подключён к реальному прогрессу)

#### 2.3.2. «✨ Готово» toast
- **Паттерн:** после успешной генерации — toast «✨ Готово, 5 операций создано»
- **Почему хорошо:** confirmation без перезагрузки страницы
- **Наше применение:** после quickGenerate() — toast с количеством операций

#### 2.3.3. Side-by-side preview
- **Паттерн:** слева промпт, справа live preview
- **Почему хорошо:** пользователь сразу видит результат
- **Наше применение:** на странице генерации ТК — слева промпт/вопросы AI, справа preview маршрута

### 2.4. Notion AI — inline AI в документе

**Паттерн:** AI-кнопка в toolbar → выделяешь текст → AI помогает
- **Наше применение:** когда технолог редактирует операцию — кнопка «🤖 Улучшить формулировку»

### 2.5. GitHub Copilot — «невидимый» AI

**Паттерн:** AI-suggestion появляется как ghost-text, пользователь не видит «AI-режим»
- **Наше применение:** при вводе материала — подсказка «Сталь 09Г2С?» (autocomplete)

## 4. Сводка «что НЕ делаем» (антипаттерны)

| Антипаттерн | Источник | Почему плохо |
|---|---|---|
| «AI-режим» как отдельный экран | Старые чат-боты | Пользователь уходит из контекста работы |
| 1С-стиль с длинными формами | ИИ-Технолог | Скучно, не вызывает доверия у молодых |
| «Сначала вопросы, потом генерация» (обязательно) | Ранние AI-инструменты | 80% случаев вопросы не нужны |
| Модальные окна с confirm на каждое действие | Старые enterprise-системы | Раздражает, замедляет |
| Inline-стили в шаблонах | Legacy код | Невозможно поддерживать, нет дизайн-системы |
| 11 кнопок в action-bar | Наш старый detail.html | Пользователь не понимает что нажать |

## 5. Наш дизайн-принцип (новая формулировка)

> **«Одна главная кнопка + свёрнутый advanced + inline-обратная связь»**
>
> - **Главное действие** — одна большая кнопка в header (always visible, всегда понятно)
> - **Advanced** — свёрнуто в `<details>`, не мешает 95% пользователей
> - **Обратная связь** — мгновенный toast, progress-bar, status-badge
> - **Inline-где возможно** — не отдельные экраны, а контекстные подсказки

Это комбинация паттернов **Linear + Cursor + v0**, адаптированная для on-premise завода.
