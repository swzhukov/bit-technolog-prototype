# ADR-0010: v0.6 prototype — 9 экранов с мок-данными для пилота 27.07.2026

**Дата:** 2026-07-21
**Статус:** Принято
**Спринт:** M31 (все 4 спринта за один заход)

## Контекст

К пилоту 27 июля 2026 (через 5 дней) нужно показать прототип v0.6:
- (a) продемонстрировать продукт команде Техинкома
- (b) решить реальные проблемы
- (c) получить финансирование
- (d) быть production-ready

Из разбора в gptunnel (см. CHANGELOG M31) и прототипа из вложения — есть 9 экранов с мок-данными.
Текущий код имеет 1 экран (главная таблица деталей) с устаревшим UI.

## Решение

**Реализовать все 4 спринта за один заход (Sprint 0+1+2+3), не откладывая на "после пилота".**

Приоритеты:
1. **Sprint 0 (подготовка)**: Mock LLM, auth, мок-данные, стиль — без этого ничего не работает.
2. **Sprint 1 (Dashboard)**: Главный экран "Мои задачи" — то что видит технолог первым.
3. **Sprint 2 (Карточка детали)**: 5 табов (Чертёж / РС / Обоснование / Доп.параметры / История) — это ядро работы технолога.
4. **Sprint 3 (Извещения/Профили/БЗ/LLM)**: Остальные экраны — для полноты картины.

## Альтернативы (которые отвергли)

1. **Делать только Dashboard + Карточку детали** (минимальный demo) — отвергли. Сергей сказал "пройдём все спринты до демонстрации".
2. **Редизайн БД (product_models, bom_links MANY-TO-MANY)** — отвергли для пилота. Слишком рискованно (26 details, 234 equipment, 5 users, drafts/versions). В ADR-0011 (после пилота).
3. **Реальный LLM** — отвергли для пилота. Ключ невалиден, время на интеграцию. Mock_llm возвращает предсказуемые ответы в правильном формате.

## Последствия

**Положительные:**
- Production deployment работает (commit 247f469), все 9 экранов возвращают 200
- Технолог может за 5 минут пройти сценарий: Dashboard → Карточка → Чертёж+Операции → РС → Извещения
- Mock LLM обеспечивает предсказуемость демо (без сетевых проблем)
- 290/290 тестов passing (ничего не сломали)

**Отрицательные:**
- `app.py` вырос с 4904 до 5500+ строк (добавлено ~600 строк)
- Новые endpoints не покрыты unit-тестами (только smoke-test)
- CSP middleware имеет warning "No response returned" — не критично, но шумно
- Login через Basic Auth (`user:pass`), а не через форму (5 пользователей в БД не задействованы в UI)
