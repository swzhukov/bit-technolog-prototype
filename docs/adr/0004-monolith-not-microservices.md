# ADR-0004: Монолит (app.py 5000 строк) вместо микросервисов

**Дата:** 2026-05-01
**Статус:** Accepted
**Контекст:** Продукт растёт — 5 модулей (rag, llm, economics, learning, drawing_recognize), 22 Python файла, 13 000 строк. Нужно решить: разбить на микросервисы или оставить монолит.

## Решение

Монолит в `app.py` + 21 модуль. Никаких микросервисов.

## Обоснование

| Фактор | Монолит ✅ | Микросервисы |
|--------|-----------|--------------|
| Команда | 1 разработчик (Mavis) | Нужна команда 3-5 |
| On-premise | Один процесс = один systemd unit | Нужно оркестрировать |
| Latency между сервисами | 0 ms (in-process) | 5-50 ms (HTTP) |
| Operational cost | Один deploy | 5+ deploy'ов, service mesh |
| Тестирование | Один pytest | Distributed tracing |
| Когнитивная нагрузка | Вижу всё сразу | Нужно помнить архитектуру |

## Что НЕ выносим в отдельные сервисы

- **RAG** (rag.py) — вызов частый, нужна скорость, нет смысла в отдельном сервисе
- **LLM-клиент** (llm.py) — синхронный вызов, dead simple
- **Auth** (auth.py) — middleware, должен быть в процессе
- **OCR** (drawing_recognize.py) — CPU-bound, но запускается редко, можно subprocess

## Что МОЖНО вынести (когда понадобится)

- **OCR** в отдельный процесс (если LLM-вызовы начнут конкурировать за CPU)
- **Pilot metrics** (learning.py) — когда пилот станет multi-tenant
- **Notifier** (notify.py) — если Telegram перестанет справляться

## Когда пересматривать

- Если команда вырастет до 3+ разработчиков
- Если модулей станет >30
- Если понадобится независимое масштабирование (например, OCR под нагрузкой)
- Если multi-tenant (несколько заводов на одной инстанции)
