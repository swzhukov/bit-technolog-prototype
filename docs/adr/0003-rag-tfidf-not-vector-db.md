# ADR-0003: RAG через TF-IDF вместо vector DB

**Дата:** 2026-04-25
**Статус:** Accepted
**Контекст:** Нужен поиск похожих ТК для few-shot примеров. RAG помогает LLM генерировать более точные ТК на основе существующих.

## Решение

TF-IDF (scikit-learn) + cosine similarity. Лемматизация через pymorphy2. Маппинг синонимов (Ст3↔сталь3, MIG↔полуавтсварка).

## Обоснование

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **TF-IDF** ✅ | On-prem, zero infrastructure, мгновенный поиск, прозрачный | Не понимает синонимы (но pymorphy2 + словарь смягчает) |
| Sentence-transformers | Понимает семантику | Нужна GPU, 400 MB модели, медленнее |
| OpenAI embeddings | Лучшее качество | Данные уходят за рубеж |
| Chroma/Weaviate | Production vector DB | Нужен отдельный сервис, бэкап, мониторинг |
| Qdrant | Rust, быстрый | Overkill для 50-100 ТК |

## Последствия

**Плюсы:**
- Ноль инфраструктуры (модуль rag.py + .rag/ pickle)
- Полностью on-prem
- Быстрый (<100ms на поиск)
- Прозрачный (можно посмотреть важные слова)

**Минусы:**
- Качество хуже embeddings на сложных запросах
- Не работает на 50+ ТК (cold start)

## Стратегия синонимов

В `rag.py:_SYNONYMS` словарь ~50 записей:
- Марки сталей (Ст3 ↔ сталь3 ↔ ст3сп)
- Сварка (MIG ↔ полуавтсварка ↔ MAG)
- Оборудование (кедр ↔ инверторсварка)
- Заготовки (лист ↔ листовой, пруток ↔ сортовой)

Это смягчает главный минус TF-IDF.

## Когда пересматривать

- Когда ТК станет >100 и accuracy упадёт
- Если пилот покажет что LLM не использует похожие ТК
- Если появится on-prem embedding (например, FRIDA от Сбера)
