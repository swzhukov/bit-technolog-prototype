## 2. ПРЯМЫЕ ДЕЙСТВИЯ (Категория A)

### A1. Создание новой детали

**Триггер:** Получил от конструктора комплект КД (чертёж + спецификацию), новая деталь в заказе, деталь отсутствует в системе.

**Предусловия:**
- Залогинен как `technologist` (или admin/main_technologist)
- Есть КД от конструктора (знаем обозначение, наименование, материал, массу)
- Справочники материалов/изделий заполнены

**Шаги:**
1. `GET /products` → видит 51 изделие, кнопки "Карточка →"
2. Если пустой выдачи — кнопка "＋ Новая деталь"; если не пустой — через UI нет ссылки (только "Карточка →" и фильтры). **Вариант 1:** через URL `GET /details/new`
3. Видит форму с 10 полями (2 колонки)
4. Заполняет обязательные:
   - Обозначение (например `ЛМША.301314.010`) — autofocus
   - Наименование (например `Упор продольный`)
   - Уровень (select: Деталь/Сборка/Изделие/Покупное/Составная)
5. Заполняет опциональные:
   - Тип (текст)
   - Материал (select из 18)
   - Изделие (select из 5)
   - Способ изготовления (select: make/buy/coop_da/coop_full)
   - Масса, кг (number, валидация)
   - Номер чертежа
   - ref_1c (UUID)
6. Нажимает "Создать деталь" (submit)
7. `POST /details/new` → INSERT в `items` + INSERT в `history`
8. **303 redirect** на `/detail/{new_id}`

**Ожидаемый результат:**
- Создан `items` row с новым id (52+)
- Создан `history` row (entity_type='item', action='create', user='tarrietsky')
- Страница детали открывается со статусом "Нет ТК"

**Пост-условия:**
- Можно сразу нажать "Сгенерировать ТК"
- Новый item появился в `/products`

**Время:** 1-2 минуты

### A2. Генерация ТК (AI)

**Триггер:** Деталь создана, ТК нет (`/detail/{id}` показывает "Нет ТК")

**Предусловия:**
- Залогинен как `technologist`
- Деталь существует, принадлежит этому технологу (или любой, если уровень < admin)
- Деталь НЕ покупная (`sourcing != 'buy'`) — иначе 400 (см. B5)

**Шаги:**
1. `GET /detail/{id}` → видит кнопку "Сгенерировать ТК" (только если `tech_card is None`)
2. Клик → `GET /items/{id}/generate` → форма с описанием и кнопкой "Сгенерировать черновик"
3. Нажимает кнопку → `POST /items/{id}/generate`
4. Видит progress overlay (0→90% за 25 сек)
5. **24 сек LLM + RAG** (фон)
6. Автоматический redirect на `/detail/{id}` (или с flash)
7. Видит:
   - Список операций (5-10)
   - Светофор норм: подтверждено / похоже / предположение AI
   - Кнопка "Аналоги" под каждой операцией (раскрывает top-N)
   - Inline-edit поля: Наименование, Тшт, Тпз
   - Кнопки "Утвердить и в эталоны" (для main_tech+), "Перегенерировать", "Экспорт в 1С"

**Ожидаемый результат:**
- Создан `tech_cards` row (status='draft', version=1)
- Создано N `operations` (5-10 строк)
- Записано в `edits` (audit-trail)
- LLM call в `llm_calls` (для метрик)

**Пост-условия:**
- Inline-edit доступен
- Кнопка "Аналоги" раскрывает top-N эталонов
- Можно подтвердить операции
- Можно отправить на утверждение (но это уже main_tech)

**Время:** 30-40 секунд (24 сек LLM + UI)

---

### A3. Inline-edit операции

**Триггер:** Операция создана, нужно скорректировать норму или название

**Предусловия:**
- Залогинен как `technologist`
- ТК существует, status != 'approved' (можно редактировать черновик)
- Поля в шаблоне помечены `class="editable"` (только name, time_per_unit_min, time_setup_min)

**Шаги:**
1. `GET /detail/{id}` → вкладка "Маршрут"
2. Видит таблицу операций
3. Клик на название операции (или Тшт, Тпз) → появляется `<input>`
4. Вводит новое значение
5. Enter или blur → `POST /api/operations/{id}/update` (JSON `{field, value}`)
6. Сервер валидирует field (whitelist) и value (тип)
7. UPDATE в `operations`
8. INSERT в `edits` (audit-trail: old_value, new_value, user, reason)
9. UI обновляется без reload (in-place)

**Ожидаемый результат:**
- `operations` row обновлён
- `edits` row создан
- Светофор пересчитан (если менялось time_per_unit_min)

**Пост-условия:**
- Можно продолжить редактирование
- История изменений в `/detail/{id}#history`

**Время:** 5-10 секунд на операцию

### A4. Просмотр аналогов (обоснование)

**Триггер:** Операция имеет жёлтый/красный статус ("похоже на правду" / "предположение AI"), нужны доказательства

**Предусловия:**
- ТК сгенерирована
- Операция имеет `ev.analogs` (есть эталонные похожие)

**Шаги:**
1. `GET /detail/{id}` → вкладка "Маршрут"
2. Видит таблицу операций с колонкой "Доказательство" (badge)
3. Нажимает "Аналоги" в строке операции
4. JavaScript раскрывает `<tr id="ev-{id}">` под этой строкой
5. Видит таблицу:
   - Эталон (обозначение)
   - Операция (название из эталона)
   - Сходство (%, с цветом: зелёный ≥90, жёлтый ≥50, красный <50)
   - Тшт, мин (из эталона)
   - Причина (почему выбран как аналог)
6. Сравнивает с текущей операцией

**Ожидаемый результат:**
- Решение: оставить, скорректировать, отклонить

**Пост-условия:**
- Inline-edit если нужна корректировка
- Кнопка "Подтвердить" если норма верна

**Время:** 1-2 минуты на операцию

---

### A5. Подтверждение нормы (Подтвердить)

**Триггер:** Операция проверена по аналогам, норма верна

**Предусловия:**
- Залогинен как `technologist`
- ТК существует, status != 'approved'

**Шаги:**
1. `GET /detail/{id}` → вкладка "Маршрут"
2. Нажимает "Подтвердить" в строке операции
3. Появляется inline-input с текущим значением + кнопки "OK"/"Отмена"
4. Меняет значение (или оставляет) → Enter или "OK"
5. `POST /api/operations/{id}/confirm` (JSON `{new_time}`) или `?new_time=...` в URL
6. UPDATE `operations` (time_per_unit_min = new_time)
7. INSERT в `edits` (reason='Подтверждение в UI')
8. INSERT в `work_history` (audit-trail подтверждения)
9. Метрика: `record_green_pct('all')` → запись в `pilot_metrics`
10. Location.reload() → обновлённый светофор

**Ожидаемый результат:**
- Операция помечена как подтверждённая
- Светофор: зелёный dot увеличился
- История показывает "Подтверждение в UI"

**Пост-условия:**
- Когда все операции зелёные → можно отправить на утверждение

**Время:** 5-10 секунд на операцию

### A6. Перегенерация ТК

**Триггер:** ТК не устраивает (плохие операции, не те материалы)

**Предусловия:**
- Залогинен как `technologist`
- ТК существует, status != 'approved' (иначе кнопка скрыта)

**Шаги:**
1. `GET /detail/{id}` → видит кнопку "Перегенерировать" (только если ТК не утверждена)
2. Клик → JS: `POST /api/tech-cards/{tc_id}/regenerate` (XRW)
3. Сервер: новый LLM call (mock пока)
4. **НЕ создаёт** новую tech_card (по коду — возвращает `llm_response` без INSERT)
5. UI: alert с результатом или reload

**Ожидаемый результат:**
- Возможно: новая версия ТК (зависит от реализации)

**Пост-условия:**
- Если качество не улучшилось → manual inline-edit

**Время:** 24 сек (LLM)

### A7. Экспорт РС в 1С:ERP

**Триггер:** ТК утверждена, нужно передать в производство

**Предусловия:**
- Залогинен как `technologist`
- ТК утверждена (`tech_card.is_approved = True`)
- LLM не на пике нагрузки (или это mock)

**Шаги:**
1. `GET /detail/{id}` → видит кнопку "Экспорт в 1С"
2. Клик → JS: `POST /api/items/{item_id}/export-to-1c` (XRW)
3. Сервер: генерирует RS через `RSFactory`, пишет XML в `data/one_c_exchange/out/RS_{designation}_{tc_id}.xml`
4. INSERT в `change_notices` (для аудита)
5. UI: alert с путём к файлу
6. Переходит на `/rs` → видит новый файл в списке
7. Клик "Скачать XML" → `/api/rs/download/{filename}` → скачивает файл
8. Открывает 1С:ERP → загружает XML

**Ожидаемый результат:**
- XML файл в `data/one_c_exchange/out/`
- Запись в `change_notices` (audit)

**Пост-условия:**
- В 1С:ERP создана РС

**Время:** 5 секунд (XML) + ручная загрузка в 1С

### A8. Создание извещения об изменении

**Триггер:** Получил сигнал об изменении (от конструктора, ОТК, изменения в эталоне)

**Предусловия:**
- Залогинен как `technologist`
- Знает: номер извещения, дату, основание, причину, затронутую деталь

**Шаги:**
1. `GET /notices` → видит список, кнопка "Создать извещение"
2. Клик → `GET /notices/new` → форма
3. Или из карточки детали: кнопка "+ Извещение" → `GET /notices/new?item_id={id}` (предзаполнение)
4. Заполняет:
   - Номер (например `И-2026-099`)
   - Дата (input date)
   - Основание (текст, опционально)
   - Причина (текст, required)
   - Описание (textarea, опционально)
   - Обозначение затронутой детали (например `ЛМША.301314.010`, required)
   - Автор (по умолчанию "Технолог")
5. Нажимает "Создать и обработать" → `POST /notices/new`
6. INSERT в `change_notices`, status='open'
7. **303 redirect** на `/notices/{new_id}`
8. На карточке извещения нажимает "Сгенерировать diff" (lazy, 24 сек)
9. AI возвращает `affected_operations` + `recommendation`
10. Выбирает решение: "Принять AI" / "Ручная проверка" / "Отклонить"
11. `POST /notices/{id}/resolve` с decision + notes
12. Если "Принять AI" → пересчёт РС

**Ожидаемый результат:**
- `change_notices` row создан
- AI diff сгенерирован
- РС пересчитан (если "Принять AI")

**Пост-условия:**
- Извещение в списке со статусом "Решено"
- В 1С:ERP обновлены РС

**Время:** 5 минут (с LLM)

## 3. ОТКЛОНЕНИЯ И КОРРЕКТИРОВКИ (Категория B)

### B1. AI вернул мусор / нерелевантные операции

**Условие:** После A2 (генерация) операции выглядят неправильно (не те материалы, не те профессии, нерелевантные)

**Симптомы:** Все операции с красным dot (предположение AI), пустые аналоги

**Действия технолога:**
1. Проверить эталонную базу знаний `/knowledge` — есть ли подходящие эталоны
2. Если эталоны есть, но RAG не нашёл — проверить вручную (открыть /detail/аналога)
3. Если эталонов нет → A6 (перегенерация) — может повезёт
4. Если перегенерация не помогла → A3 (inline-edit всех операций вручную)
5. Если нужен новый эталон → эскалация на main_technologist

**Восстановление:** После inline-edit → A5 (подтверждение)

---

### B2. Не нашлось эталона в RAG (для новой детали)

**Условие:** Деталь новая, в эталонной базе нет похожих

**Симптомы:** Все операции с красным dot, "Аналоги" пусто

**Действия:**
1. Проверить `/knowledge` — есть ли там что-то для product_type этой детали
2. Если нет — это нормально, для новых деталей
3. Inline-edit всех операций по своему опыту
4. Подтвердить каждую
5. **Создать эталон** (через main_technologist → утверждение) — для будущих аналогов

**Восстановление:** После утверждения ТК → новая запись в `etalons` → улучшение RAG

---

### B3. Конфликт версий при правке (другой технолог меняет ту же ТК)

**Условие:** Технолог A открыл карточку детали, технолог B её редактирует, A сохраняет

**Симптомы:** В текущей версии БИТ.Технолог **конфликта нет** (нет versioning, последний write выигрывает)

**Действия:**
1. Сохранить → UPDATE применяется
2. Если A и B делают разные изменения, **потеряется работа B**
3. Эскалация: договориться с B о разделении работы

**Восстановление:** B повторяет свои изменения. Sprint 6: версионирование.

---

### B4. Не хватает материала / оборудования в справочнике

**Условие:** При выборе материала в форме /details/new или в эталоне — нужного нет

**Симптомы:** В select нет нужного значения, эскалация к admin

**Действия технолога:**
1. Выбирает ближайший аналог из справочника
2. Заполняет поле "Материал" аналогом
3. Создаёт задачу admin'у на добавление нового материала (вне системы)
4. После добавления — повторно создаёт ТК с правильным материалом (или inline-edit)

**Восстановление:** После добавления материала в справочник → новый цикл

---

### B5. Попытка сгенерировать ТК для покупного изделия

**Условие:** Деталь имеет `sourcing = 'buy'`

**Симптомы:** `POST /items/{id}/generate` → 400 "покупное" (после M36-fix6)

**Действия:**
1. Понять что покупное не должно иметь ТК (покупаем у поставщика)
2. Не генерировать ТК
3. Только inline-edit если деталь ошибочно помечена как `buy`

**Восстановление:** Изменить `sourcing` через admin или main_tech (в UI нет)

---

### B6. Исчерпан лимит AI-генераций (5/мин)

**Условие:** Технолог нажал "Сгенерировать ТК" 5 раз за минуту

**Симптомы:** 6-й запрос → 429 + `Retry-After: <sec>`

**Действия:**
1. Подождать `Retry-After` секунд (60 сек окно)
2. Или продолжить работу без AI (просмотр дашборда, извещений)
3. Повторить генерацию

**Восстановление:** Через минуту окно сбрасывается

---

### B7. LLM провайдер недоступен (1bitai.ru упал)

**Условие:** 1bitai.ru не отвечает, или API key невалидный

**Симптомы:** `POST /items/{id}/generate` → 30-60 сек → 500 error

**Действия:**
1. В логах `/var/log/bit-technolog.err.log` — `LLMTimeout` или `LLMAuthError`
2. Автоматически используется MockLLMProvider (fallback в коде)
3. UI показывает результат (возможно хуже чем LLM)
4. Эскалация на admin (настройка LLM)

**Восстановление:** Когда 1bitai.ru восстановится — следующая генерация будет через реальный LLM

---

### B8. Дубликат обозначения при создании

**Условие:** Технолог создаёт деталь с обозначением, которое уже есть

**Симптомы:** `POST /details/new` → 400 "Деталь с обозначением «X» уже существует"

**Действия:**
1. Вернуться на форму (form_data сохранён)
2. Проверить в `/products` — возможно деталь есть
3. Если это та же деталь — открыть `/detail/{id}` существующей
4. Если новая деталь с дубликатом — изменить обозначение

**Восстановление:** После исправления → A1 успешно

---

### B9. Невалидные данные в форме создания

**Условие:** Одно из полей невалидно (масса не число, level вне списка, FK не найден)

**Симптомы:** 400 + сообщение в форме, form_data сохранён

**Действия:**
1. Прочитать сообщение об ошибке
2. Исправить поле
3. Submit снова

**Восстановление:** После исправления → A1 успешно

---

### B10. Не хватает прав (например, на утверждение ТК)

**Условие:** Технолог нажимает "Утвердить и в эталоны"

**Симптомы:** Кнопка скрыта в UI (для technologist), но если через API → 403

**Действия:**
1. Видит что кнопки нет
2. Эскалирует на main_technologist или admin
3. Может сам inline-edit + подтвердить все операции → ТК готова к утверждению

**Восстановление:** main_tech заходит и утверждает

---

### B11. Не могу изменить утверждённую ТК

**Условие:** Технолог открывает карточку с утверждённой ТК

**Симптомы:** Кнопки "Перегенерировать" / inline-edit скрыты (есть только "Экспорт в 1С")

**Действия:**
1. Понять что ТК утверждена — нельзя менять
2. Если нужны изменения — создать **извещение** об изменении (A8)
3. Через извещение → пересмотр РС → новая версия ТК

**Восстановление:** Новая версия ТК через извещение

---

### B12. Потеря сессии во время работы

**Условие:** Технолог работал час, вдруг logout / 401

**Симптомы:** `fetch` → 401 → `window.location.href = '/login'`

**Действия:**
1. Login заново
2. Все несохранённые данные потеряны (форма не сохраняется)
3. Inline-edit (был сохранён на сервере) — сохранён
4. Открыть `/detail/{id}` — ТК на месте

**Восстановление:** Re-login → продолжить
